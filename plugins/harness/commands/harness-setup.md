# `/harness-setup` — Harness installation and integrity

This skill consolidates all harness-lifecycle operations that used to live in
separate commands. It is domain-neutral; use `harness.config.json` at the
project root to encode any project-specific values.

## Subcommands

| Subcommand | Purpose |
|-----------|---------|
| `/harness-setup init`   | Initialize a fresh `harness.config.json` and scaffold `CLAUDE.md` if missing |
| `/harness-setup check`  | Verify the plugin is correctly installed (files, manifest, hooks) |
| `/harness-setup doctor` | Print environment diagnostics (Node version, codex plugin, `CLAUDE_PLUGIN_ROOT`, …) |
| `/harness-setup build`  | Rebuild the TypeScript core (`npm run build`) |
| `/harness-setup localize` | Interactively tune `harness.config.json` for the current project |

## `init` — fresh project bootstrap

1. Read `harness.config.json` at the project root. If missing, create it from
   the schema defaults (empty `protectedDirectories`, standard
   `protectedEnvVarNames`, `.env` suffix). Use the `harness init` CLI if it is
   on PATH, or copy from `template/.claude/harness.config.json.tmpl`.
2. If `CLAUDE.md` does not exist, copy the template from
   `template/.claude/CLAUDE.md.tmpl` and fill in `{{PROJECT_NAME}}` from the
   top-level directory name.
3. Ensure `.gitignore` excludes `.claude/logs/`, `.claude/state/`,
   `.claude/worktrees/`, and `harness.config.json.local`.
4. Print a short summary of the generated files.

The files generated by `init` are:

```
<project-root>/
├── CLAUDE.md
├── harness.config.json
├── Plans.md
└── .claude/
    ├── settings.local.json     # permissions
    └── rules/                  # copied from template/.claude/rules/
```

## `check` — installation integrity

Run the `harness check` CLI binary or perform the equivalent in-session.
Verify that the following files exist inside the installed plugin:

- `.claude-plugin/plugin.json` — manifest
- `hooks/hooks.json` — hook registration
- `scripts/hook-dispatcher.mjs` — dispatcher
- `core/dist/index.js` — compiled core entry point

And that the following agent definitions exist (v3 layout):

- `agents/worker.md`
- `agents/reviewer.md`
- `agents/scaffolder.md`
- `agents/security-auditor.md`
- `agents/codex-sync.md`

And that the 5 verb skills exist:

- `commands/harness-plan.md`
- `commands/harness-work.md`
- `commands/harness-review.md`
- `commands/harness-release.md`
- `commands/harness-setup.md`

Report each file as OK / MISSING. Exit non-zero if any are missing.

### Expected layout (v3)

```
plugins/harness/
├── .claude-plugin/plugin.json
├── hooks/hooks.json
├── scripts/
│   ├── hook-dispatcher.mjs
```

## `doctor` — environment diagnostics

Print the following information:

- `harness` version (from `plugin.json`)
- Node.js version (`process.version`)
- Platform and architecture
- `CLAUDE_PLUGIN_ROOT` and `CLAUDE_PROJECT_DIR` env vars
- Resolved core build path and modification time
- Whether the codex plugin is installed at `~/.claude/plugins/cache/openai-codex/codex/`

Use this when something is wrong and you need a starting point for
troubleshooting.

## `build` — rebuild the core

```bash
cd plugins/harness/core
npm install
npm run build
```

Produces `core/dist/`. Run this after any change to `core/src/`.

Run `npm test` afterwards to confirm all R01–R13 rules, the state store,
tampering detector, and config loader are green.

## `localize` — tune for the current project

Guide the user through populating `harness.config.json`:

- `projectName` — used in messages
- `language` — `en` or `ja`
- `protectedDirectories` — directories R10 refuses to delete (e.g.
  datasets, generated artifacts)
- `protectedEnvVarNames` — additions to the default secret-name list
- `protectedFileSuffixes` — in addition to `.env`
- `codex.enabled` — whether to surface the `codex-sync` agent
- `workMode` — per-project bypass defaults for R05 / R06
- `tampering.severity` — `approve` (warn), `ask` (confirm), `deny` (block)

Write the file and run `/harness-setup check` afterwards.

## Prohibited actions

- Never modify files in `protectedDirectories` directly
- Never write to `.env` or any path matching `protectedFileSuffixes`
- Never skip this check in CI — `harness check` is cheap

## Agents available after setup

| Agent | Role |
|-------|------|
| `worker`     | Implements changes, runs self-review, verifies, commits |
| `reviewer`   | Read-only, multi-angle code / plan / scope review |
| `scaffolder` | Keeps docs, `CLAUDE.md`, and `harness.config.json` in sync |
| `security-auditor` | Audits for credential leakage, injection, permissions |
| `codex-sync` | Synchronous wrapper around the Codex CLI companion |
