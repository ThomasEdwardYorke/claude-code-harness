## [Unreleased]

### Fixed

- **Single `Task` tool drift removal across 6 commands + 5 agents + body-text references (Codex B CRITICAL C.1 follow-up)** вАФ official Claude Code tools catalog ([tools-reference](https://code.claude.com/docs/en/tools-reference)) does **not** contain a single `Task` tool; subagent spawning is performed by `Agent`, and task-list lifecycle is split into `TaskCreate` / `TaskGet` / `TaskList` / `TaskStop` / `TaskUpdate` (`TaskOutput` is marked *(Deprecated)* in the same table). Six commands (`harness-plan` / `harness-review` / `harness-work` / `parallel-worktree` / `pseudo-coderabbit-loop` / `tdd-implement`) previously listed `"Task"` in `allowed-tools`, which official runtime parses as non-tool (silent ignore) and breaks `claude plugin validate` on stricter future releases. Five agent frontmatters (`worker` / `scaffolder` / `reviewer` / `security-auditor` / `codex-sync`) previously used `disallowedTools: [..., Task]`, which was a no-op and failed to enforce the intended "no nested subagent spawning" invariant вАФ now corrected to `disallowedTools: [..., Agent]`. Body-text drift in 3 command specs (`harness-work.md` / `parallel-worktree.md` / `pseudo-coderabbit-loop.md`) referring to `Task tool` / `Task гГДгГЉгГЂ` / `Task({...})` pseudocode / `Parallel (Task tool)` mode name is rewritten to `Agent tool` / `Agent({...})` / `Parallel (Agent tool)` to match shipped reality. The `harness-work.md` dispatcher mode identifier moves from `parallel-task-tool` to `parallel-agent-tool` for the same reason. `harness-setup.md` config-field description (`work.maxParallel`) also drops the legacy `Task`-tool wording.
- **Added `content-integrity.test.ts` regression guards (new `describe` block, 3 √Ч subject + 4 √Ч self-test)** вАФ covers command `allowed-tools` / agent `tools` / agent `disallowedTools`. Each subject block enforces both (a) single `Task` is absent and (b) every listed value is in the official 35-tool catalog (`OFFICIAL_TOOL_NAMES`) or is a dynamic MCP tool (`mcp__server__action`). The parser `parseToolListField` handles all 5 officially documented frontmatter formats per slash-commands spec ("space-separated string or YAML list"): JSON-style array, YAML flow array (quoted / unquoted / multi-line), YAML block array (`- item`), space-separated string, CSV string, and granular permission specifiers (`Bash(git add *)`). Helper self-tests (`stripGranularSpec` / `parseToolListField` / `isMcpDynamicTool`) guard against future refactor regression. Source of the 35-tool whitelist is the official [tools-reference](https://code.claude.com/docs/en/tools-reference) table fetched on 2026-04-23 and independently verified by Codex Worker A / Worker B / Reviewer (three independent agents, consensus on `Task` being absent). Test count moved from 1836 вЖТ 1884 (+48 effective it-cases, net 12 reduction due to DRY helper combining "Task forbidden" and "whitelist conformant" into a single per-subject assertion).

## [0.3.0] - 2026-04-23

### Added

- **`WorktreeCreate` hook blocking protocol production (Phase ќЇ-2)** вАФ previously a scaffold-only handler deferred until `isolation: worktree` coordination design. Now registered in `hooks.json` (`command` type, timeout 120s) and implemented as a production blocking hook per the official contract (https://code.claude.com/docs/en/hooks): a `command` hook prints the absolute worktree path to stdout on success, and any non-zero exit causes worktree creation to fail (blocking semantics вАФ unlike other hooks there is no special exit-2 behavior). `handleWorktreeCreate` now executes real `git worktree add <path> -b harness-wt/<name>` with: (a) strict name validation (allowlist `[A-Za-z0-9_][A-Za-z0-9_.-]{0,63}` + git refname rule enforcement: no consecutive `..`, no trailing `.`, no `.lock` suffix, no path separators / `.` prefix / shell metachars), (b) `<parent-of-cwd>/<basename-of-cwd>-wt-<name>` sibling path convention (aligns with `/parallel-worktree` manual layout when both use this prefix), (c) `harness-wt/` branch prefix to avoid namespace collision with user branches, (d) idempotent behavior (2nd call with same name + branch returns existing path via porcelain stanza parsing of both `worktree` and `branch refs/heads/...`), (e) `realpathSync.native` normalization of `projectRoot` on entry (macOS `/var` вЖФ `/private/var` symlink diff and Windows 8.3 short-name `RUNNER~1` вЖФ `runneradmin` both eliminated), (f) explicit split of repository presence check (`isGitRepository`) vs unborn-HEAD check (`gitHasCommit`) for clearer reason strings, (g) early newline/CR reject on `projectRoot` to prevent downstream stdout/stderr corruption, (h) post-race recheck after `git worktree add` failure to handle concurrent create. No orphan-branch fallback retry: if `-b` fails because `harness-wt/<name>` already exists, the hook surfaces the git error and the user is expected to clean up stale branches (`git branch -D harness-wt/<name>`) before retrying. `HookResult` gains optional `worktreePath` / `continue` / `stopReason` / `suppressOutput` fields (backward-compatible, used only by the new WorktreeCreate path). `index.ts main()` gains a `worktree-create` branch that writes raw stdout path on success or exits 1 on failure per the blocking contract (via `process.exitCode = 1` so stderr/stdout drain before termination). The existing `content-integrity.test.ts` Phase ќЈ P0-ќЇ guard that asserted `WorktreeCreate` *not* registered is flipped to assert it *is* registered with `timeout: 120`, plus the scaffold self-identification guard is replaced with a production-blocking-protocol guard (requires `git worktree add` + blocking / exit / non-zero / worktreePath wording and forbids any `scaffold ... deferred` combination). `marketplace.json` plugin description is updated to **12 lifecycle hook events** (was 11). The `isolation: worktree` frontmatter is intentionally NOT added to any agent in this PR; that migration is deferred pending `/parallel-worktree` co-existence design (preventing nested-worktree interference). Red/Green TDD discipline (self-review + Codex adversarial review √Ч2 rounds + pseudo-CodeRabbit review + 7 real CodeRabbit review rounds + Codex second-opinion pre-merge all closed, every actionable finding addressed): 20 net new tests across 3 files вАФ 16 handler unit tests in `worktree-lifecycle.test.ts` (injection defense + git-refname rules, idempotent reuse + post-race recheck, realpath normalization, isolated-copy check, unborn-HEAD diagnostic, non-git-repo vs unborn-HEAD separation, newline/CR projectRoot reject, branch-mismatch non-reuse, subdirectory-safe repo-root resolution, `.worktreeinclude` replication with path-traversal reject); 1 route-level dispatcher test in `index.test.ts` for `worktreePath` propagation (replaces the old scaffold test; fail-open reason path is covered by the adjacent test); 2 e2e child-process tests in `index.test.ts` asserting raw-stdout path on success plus non-zero exit on failure; 1 types test in `types.test.ts` covering optional field compile-time compatibility. 16 + 1 + 2 + 1 = 20. Test count 1836 вЖТ 1856.

Additional hardening driven by Codex second-opinion (pre-merge) review:
- **Subdirectory-safe repo root resolution**: `cwd` hook payload is Claude Code's CWD, not necessarily repo top-level. Previously a subdirectory invocation of the hook placed the sibling worktree *inside* the repo tree and broke idempotency (path+branch mismatch). Now uses `git rev-parse --show-toplevel` to resolve the actual repo root and bases `<parent>/<basename>-wt-<name>` on that.
- **`.worktreeinclude` replication**: registering `WorktreeCreate` fully replaces Claude Code's default `git worktree add`, which silently dropped copying of files listed in `.worktreeinclude` (e.g. `.env`). The handler now reads the file on success, validates each entry against path traversal (no `..`, no absolute paths, prefix-match on both repo root and worktree root), and `cpSync`s safely into the new worktree. Failures are non-fatal (worktree creation is already done) and surfaced via `[worktreeinclude-copied]` / `[worktreeinclude-skipped]` / `[worktreeinclude-failed]` notes in `additionalContext`.
- **`-z` porcelain output + prunable/existsSync filter**: `findExistingWorktree` now uses `git worktree list --porcelain -z` (NUL-separated) so paths with embedded newlines don't corrupt parsing, skips `prunable` stanzas, and verifies `existsSync(path)` before treating a stanza as a reuse candidate.
- **Dispatcher-level protocol fix** (`scripts/hook-dispatcher.mjs`): the generic `failOpen()` used to print JSON and exit 0 for every hook type, which would have corrupted the WorktreeCreate raw-stdout contract if the dispatcher itself failed (e.g. `CLAUDE_PLUGIN_ROOT` unset, `core/dist` missing). A `worktree-create` special case now writes the failure to stderr and exits 1, matching the blocking hook contract end-to-end.

### Added

- **`session-handoff` skill `check` v2 follow-up (PR #6)** вАФ explicit **full-context ingestion** in Gate 2 description (`Read` loads `current.md` + `backlog.md` in full to Claude context, report is summary-only but content is query-able), new **Anti-pattern #10** (re-reading after `check` is redundant, exception is compaction), new staleness signal **S-13** (backlog.md 150+ lines WARN / 200+ lines FAIL, using Gate 2 Context loaded rather than separate `wc -l`, classified under Gate 1 Structural), Output Template adds `Context loaded: <N> lines (current: {X}, backlog: {Y})` Summary field for re-bloat visibility, and `Forbidden` section compressed from 8 bullets to 3 categories (no information loss). Skill size kept under 500 lines per Anthropic SKILL.md guideline.
- **`session-handoff` skill `check` v2 вАФ 3-gate architecture (structural + content comprehension + rehydration synthesis)**. The `check` subcommand was extended from a structural-only linter (v1) to a three-gate read-only verification: **Gate 1 Structural Integrity** (existence, line limits, priority labels, append-only, naming conventions вАФ unchanged from v1), **Gate 2 Content Comprehension** (reads `current.md` / `backlog.md` via Read tool, extracts Latest state / Top priority / Quick-start / Pointers, cross-checks with `git log --oneline -5`), **Gate 3 Understanding Synthesis** (evaluates 12 staleness signals `S-01`гАЬ`S-12` with severity, produces a 3-level rehydration verdict `PASS (Ready)` / `WARN (Partial)` / `FAIL (Stale)` plus the special `INIT_REQUIRED` for first-time use). - **`session-handoff` skill (workflow commands: 7 вЖТ 8; total commands: 12 вЖТ 13)** вАФ new `plugins/harness/commands/session-handoff.md` codifies a 3-layer session-handoff document structure (`<project>-current.md` bird's-eye index + `<project>-backlog.md` + `<project>-design-decisions.md` + `archive/session-<YYYY-MM-DD>-<slug>.md` per-session records). - **Phase ќЈ P0-ќЇ вАФ `WorktreeRemove` hook handler (non-blocking observability)**. New `plugins/harness/core/src/hooks/worktree-lifecycle.ts` implements `handleWorktreeRemove`, registered in `plugins/harness/hooks/hooks.json` (dispatch arg `worktree-remove`, timeout 10s). - **Phase ќЈ P0-ќЇ вАФ `WorktreeCreate` handler scaffold (intentionally NOT registered in `hooks.json`)**. `handleWorktreeCreate` is exported and wired through `route()` / `main()` in `core/src/index.ts` so that Phase ќЇ-2 can drop in a protocol-compliant blocking implementation (stdout absolute-path contract + real `git worktree` creation) without rewiring the dispatch layer. A regression guard in `content-integrity.test.ts` asserts that `hooks.json` does NOT contain `WorktreeCreate` until Phase ќЇ-2 ships. The handler's doc-comment now enumerates the concrete failure mode (JSON stdout unparseable as path вЖТ `worktreePath` empty вЖТ worktree creation fails) to deter future accidental promotion.
- **Phase ќЈ P0-ќЇ regression guards** (`content-integrity.test.ts`, 5 tests): `hooks.json` must register `WorktreeRemove` and must NOT register `WorktreeCreate`; `worktree-lifecycle.ts` must export both handlers; context injection must use `additionalContext` (not `systemMessage`); the `WorktreeCreate` scaffold must self-identify with a **two-term combination** (`Phase ќЇ-2`+`deferred` OR `scaffold`+`hooks.json жЬ™зЩїйМ≤`) to resist single-word false-positive matches.

## [0.1.0] - 2026-04-16

### Added
- Initial portable release split from a private project harness.
- Guardrail rules R01вАУR13 with TypeScript implementation in `plugins/harness/core/`.
- `harness.config.json` schema and loader for per-project customization.
- Pure-JS JSON file state store (zero native dependencies).
- Plugin manifest (`plugin.json`, `marketplace.json`) for Claude Code distribution.
- 5 verb skills: `harness-plan/work/review/release/setup`.
- 3 general agents: `worker`, `reviewer`, `scaffolder`.
- Helper agents: `security-auditor`, `codex-sync`.
- E2E smoke tests that run against compiled core without requiring a Claude Code session.
- GitHub Actions workflows for CI, build, release, and smoke testing.

### Changed
- R10 (protected directory deletion) is now parameter-driven (`protectedDirectories`), no-op when empty.
- R11 (API-key-in-command) is parameter-driven (`protectedEnvVarNames`), no-op when empty.
- All guardrail error messages are now in English and dynamically constructed from config.
- `harness-setup check` now references v3 agent names (`worker`/`reviewer`/`scaffolder`).
- State storage path unified to `.claude/state/harness.json` (JSON, not SQLite).

### Removed
- `better-sqlite3` dependency (native binary) replaced with pure JS JSON store.
- Project-specific agents (`script-tester`, `test-writer`) вАФ see migration guide.
- Project-specific rule (`project-architecture.md`).

### Security
- Added explicit guidance on log sensitivity in `docs/en/security.md`.
- `.gitignore` template excludes `.claude/logs/`, `.claude/state/`, `.claude/worktrees/`.
