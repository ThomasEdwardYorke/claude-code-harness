## [Unreleased]

### Added

- **`WorktreeCreate` hook blocking protocol production (Phase Оә-2)** вҖ” previously a scaffold-only handler deferred until `isolation: worktree` coordination design. Now registered in `hooks.json` (`command` type, timeout 120s) and implemented as a production blocking hook per the official contract (https://code.claude.com/docs/en/hooks): a `command` hook prints the absolute worktree path to stdout on success, and any non-zero exit causes worktree creation to fail (blocking semantics вҖ” unlike other hooks there is no special exit-2 behavior). `handleWorktreeCreate` now executes real `git worktree add <path> -b harness-wt/<name>` with: (a) strict name validation (allowlist `[A-Za-z0-9_][A-Za-z0-9_.-]{0,63}` + git refname rule enforcement: no consecutive `..`, no trailing `.`, no `.lock` suffix, no path separators / `.` prefix / shell metachars), (b) `<parent-of-cwd>/<basename-of-cwd>-wt-<name>` sibling path convention (aligns with `/parallel-worktree` manual layout when both use this prefix), (c) `harness-wt/` branch prefix to avoid namespace collision with user branches, (d) idempotent behavior (2nd call with same name + branch returns existing path via porcelain stanza parsing of both `worktree` and `branch refs/heads/...`), (e) `realpathSync.native` normalization of `projectRoot` on entry (macOS `/var` вҶ” `/private/var` symlink diff and Windows 8.3 short-name `RUNNER~1` вҶ” `runneradmin` both eliminated), (f) explicit split of repository presence check (`isGitRepository`) vs unborn-HEAD check (`gitHasCommit`) for clearer reason strings, (g) early newline/CR reject on `projectRoot` to prevent downstream stdout/stderr corruption, (h) post-race recheck after `git worktree add` failure to handle concurrent create. No orphan-branch fallback retry: if `-b` fails because `harness-wt/<name>` already exists, the hook surfaces the git error and the user is expected to clean up stale branches (`git branch -D harness-wt/<name>`) before retrying. `HookResult` gains optional `worktreePath` / `continue` / `stopReason` / `suppressOutput` fields (backward-compatible, used only by the new WorktreeCreate path). `index.ts main()` gains a `worktree-create` branch that writes raw stdout path on success or exits 1 on failure per the blocking contract (via `process.exitCode = 1` so stderr/stdout drain before termination). The existing `content-integrity.test.ts` Phase О· P0-Оә guard that asserted `WorktreeCreate` *not* registered is flipped to assert it *is* registered with `timeout: 120`, plus the scaffold self-identification guard is replaced with a production-blocking-protocol guard (requires `git worktree add` + blocking / exit / non-zero / worktreePath wording and forbids any `scaffold ... deferred` combination). `marketplace.json` plugin description is updated to **12 lifecycle hook events** (was 11). The `isolation: worktree` frontmatter is intentionally NOT added to any agent in this PR; that migration is deferred pending `/parallel-worktree` co-existence design (preventing nested-worktree interference). Red/Green TDD discipline (self-review + Codex adversarial review Г—2 rounds + pseudo-CodeRabbit review + 7 real CodeRabbit review rounds + Codex second-opinion pre-merge all closed, every actionable finding addressed): 20 net new tests across 3 files вҖ” 16 handler unit tests in `worktree-lifecycle.test.ts` (injection defense + git-refname rules, idempotent reuse + post-race recheck, realpath normalization, isolated-copy check, unborn-HEAD diagnostic, non-git-repo vs unborn-HEAD separation, newline/CR projectRoot reject, branch-mismatch non-reuse, subdirectory-safe repo-root resolution, `.worktreeinclude` replication with path-traversal reject); 1 route-level dispatcher test in `index.test.ts` for `worktreePath` propagation (replaces the old scaffold test; fail-open reason path is covered by the adjacent test); 2 e2e child-process tests in `index.test.ts` asserting raw-stdout path on success plus non-zero exit on failure; 1 types test in `types.test.ts` covering optional field compile-time compatibility. 16 + 1 + 2 + 1 = 20. Test count 1836 вҶ’ 1856.

Additional hardening driven by Codex second-opinion (pre-merge) review:
- **Subdirectory-safe repo root resolution**: `cwd` hook payload is Claude Code's CWD, not necessarily repo top-level. Previously a subdirectory invocation of the hook placed the sibling worktree *inside* the repo tree and broke idempotency (path+branch mismatch). Now uses `git rev-parse --show-toplevel` to resolve the actual repo root and bases `<parent>/<basename>-wt-<name>` on that.
- **`.worktreeinclude` replication**: registering `WorktreeCreate` fully replaces Claude Code's default `git worktree add`, which silently dropped copying of files listed in `.worktreeinclude` (e.g. `.env`). The handler now reads the file on success, validates each entry against path traversal (no `..`, no absolute paths, prefix-match on both repo root and worktree root), and `cpSync`s safely into the new worktree. Failures are non-fatal (worktree creation is already done) and surfaced via `[worktreeinclude-copied]` / `[worktreeinclude-skipped]` / `[worktreeinclude-failed]` notes in `additionalContext`.
- **`-z` porcelain output + prunable/existsSync filter**: `findExistingWorktree` now uses `git worktree list --porcelain -z` (NUL-separated) so paths with embedded newlines don't corrupt parsing, skips `prunable` stanzas, and verifies `existsSync(path)` before treating a stanza as a reuse candidate.
- **Dispatcher-level protocol fix** (`scripts/hook-dispatcher.mjs`): the generic `failOpen()` used to print JSON and exit 0 for every hook type, which would have corrupted the WorktreeCreate raw-stdout contract if the dispatcher itself failed (e.g. `CLAUDE_PLUGIN_ROOT` unset, `core/dist` missing). A `worktree-create` special case now writes the failure to stderr and exits 1, matching the blocking hook contract end-to-end.

## [0.2.0] - 2026-04-23

> **вҡ пёҸ Breaking change for v3/v4.1 users**: The `--test-pipeline` subflow has been fully removed from `harness-work.md`. If you relied on it, migrate to a project-local skill under `.claude/skills/<project>-local-rules/references/pipeline-check.md`. See `### Removed` below.

### Added

- **`session-handoff` skill `check` v2 follow-up (PR #6)** вҖ” explicit **full-context ingestion** in Gate 2 description (`Read` loads `current.md` + `backlog.md` in full to Claude context, report is summary-only but content is query-able), new **Anti-pattern #10** (re-reading after `check` is redundant, exception is compaction), new staleness signal **S-13** (backlog.md 150+ lines WARN / 200+ lines FAIL, using Gate 2 Context loaded rather than separate `wc -l`, classified under Gate 1 Structural), Output Template adds `Context loaded: <N> lines (current: {X}, backlog: {Y})` Summary field for re-bloat visibility, and `Forbidden` section compressed from 8 bullets to 3 categories (no information loss). S-01гҖңS-13 total, Anti-patterns total 10. 5 new regression guards in `content-integrity.test.ts` (full-context wording / Context-loaded line / re-read forbidden / S-13 threshold / 10th anti-pattern existence). Skill size kept under 500 lines per Anthropic SKILL.md guideline.
- **`session-handoff` skill `check` v2 вҖ” 3-gate architecture (structural + content comprehension + rehydration synthesis)**. The `check` subcommand was extended from a structural-only linter (v1) to a three-gate read-only verification: **Gate 1 Structural Integrity** (existence, line limits, priority labels, append-only, naming conventions вҖ” unchanged from v1), **Gate 2 Content Comprehension** (reads `current.md` / `backlog.md` via Read tool, extracts Latest state / Top priority / Quick-start / Pointers, cross-checks with `git log --oneline -5`), **Gate 3 Understanding Synthesis** (evaluates 12 staleness signals `S-01`гҖң`S-12` with severity, produces a 3-level rehydration verdict `PASS (Ready)` / `WARN (Partial)` / `FAIL (Stale)` plus the special `INIT_REQUIRED` for first-time use). Includes explicit read-only `Forbidden` section, edge-case handling (git / gh unavailable, large archive sampling, markdown corruption), and a bold-heading output template that avoids `##` collision with downstream regex scanners. 7 regression guards added to `content-integrity.test.ts` covering: 3-gate keyword presence / Read+current+backlog declaration / вүҘ6 staleness signals + 4-topic coverage / 3-level verdict / code-block template / forbidden-ops enumeration / edge-case coverage (first-time + git-unavailable both required independently).
- **`session-handoff` skill (workflow commands: 7 вҶ’ 8; total commands: 12 вҶ’ 13)** вҖ” new `plugins/harness/commands/session-handoff.md` codifies a 3-layer session-handoff document structure (`<project>-current.md` bird's-eye index + `<project>-backlog.md` + `<project>-design-decisions.md` + `archive/session-<YYYY-MM-DD>-<slug>.md` per-session records). The skill is fully generic (no project-specific names), and its structure aligns with Anthropic's official [MEMORY.md][anthropic-memory] and [SKILL.md][anthropic-skills] patterns. Subcommands: `init` / `update` / `archive` / `check`. Eight anti-patterns are enumerated up-front to prevent the common regression of accumulating session logs in a single monolithic prompt. 12 regression guards added to `content-integrity.test.ts` (10 skill-specific: existence, frontmatter 4-field compliance, strict `[word|word|...]` argument-hint regex, < 500 lines per Anthropic SKILL.md guideline, `<project>` placeholder presence, four-file layout mention, update-trigger section, anti-pattern section, Anthropic-official reference citation, Japanese-imperative `description-ja`; plus 2 harness-setup integration: command-list path-form check + plugin.json вҶ” `harness-setup.md` workflow-skill-count consistency with VERB_SKILLS array drift guard).
- **Phase О· P0-Оә вҖ” `WorktreeRemove` hook handler (non-blocking observability)**. New `plugins/harness/core/src/hooks/worktree-lifecycle.ts` implements `handleWorktreeRemove`, registered in `plugins/harness/hooks/hooks.json` (dispatch arg `worktree-remove`, timeout 10s). Emits `additionalContext` covering `worktree_path` / `agent_type` / `agent_id` plus a `Plans.md` assignment-table reminder for the coordinator when the file contains known section markers. Payload values are newline-sanitised before being joined into the context to prevent fake-section injection. Fail-open via `loadConfigSafe` (shape-invalid config silently falls back to defaults). Spec basis: `docs/maintainer/research-anthropic-official-2026-04-22.md` В§ 3 (гғ•гғғгӮҜ).
- **Phase О· P0-Оә вҖ” `WorktreeCreate` handler scaffold (intentionally NOT registered in `hooks.json`)**. `handleWorktreeCreate` is exported and wired through `route()` / `main()` in `core/src/index.ts` so that Phase Оә-2 can drop in a protocol-compliant blocking implementation (stdout absolute-path contract + real `git worktree` creation) without rewiring the dispatch layer. The handler returns `approve` with a scaffold-notice `additionalContext` and never writes stdout paths. Rationale: `WorktreeCreate` **replaces** Claude Code's default `git worktree` creation entirely; registering an observability-only hook would break worktree creation (see `docs/maintainer/research-anthropic-official-2026-04-22.md` В§ 3). A regression guard in `content-integrity.test.ts` asserts that `hooks.json` does NOT contain `WorktreeCreate` until Phase Оә-2 ships. The handler's doc-comment now enumerates the concrete failure mode (JSON stdout unparseable as path вҶ’ `worktreePath` empty вҶ’ worktree creation fails) to deter future accidental promotion.
- **Phase О· P0-Оә regression guards** (`content-integrity.test.ts`, 5 tests): `hooks.json` must register `WorktreeRemove` and must NOT register `WorktreeCreate`; `worktree-lifecycle.ts` must export both handlers; context injection must use `additionalContext` (not `systemMessage`); the `WorktreeCreate` scaffold must self-identify with a **two-term combination** (`Phase Оә-2`+`deferred` OR `scaffold`+`hooks.json жңӘзҷ»йҢІ`) to resist single-word false-positive matches.
- **Route-level integration tests for both hooks** (`index.test.ts`, 3 tests) and **handler unit tests** (`worktree-lifecycle.test.ts`, new file, 14 tests including empty-string, newline-injection, and CR/CRLF edge cases). New `it` / `it.each` blocks added across the three-commit Phase О· P0-Оә series total 22 (net, across 3 test files). The project-wide vitest case count (which expands `it.each` parametrisations from many earlier test files) moved from 1685 to 1760 across the series.
- **Phase Оә вҖ” subagent `isolation` frontmatter regression guard** (`content-integrity.test.ts`). 12 tests (6 agent Г— 2 assertion) that lock in the current no-`isolation` policy and permit only the official `"worktree"` value if someone ever adds it. Background in `docs/maintainer/research-subagent-isolation-2026-04-22.md` (Codex Worker C, official-docs deep dive): the `isolation: worktree` field is supported for plugin subagents, but adding it to `worker` while `/parallel-worktree` still manages worktrees manually would create a nested-worktree interference risk. Migrating cleanly requires hook-level coordination via `WorktreeCreate` / `WorktreeRemove` (tracked as Phase Оә-2, Phase 2-3 scope).
- **Phase О» вҖ” B-2f generality guard** (`generality.test.ts`). Blocks re-introduction of the predecessor-project-specific `--test-pipeline` subflow in shipped specs. Regex `/--test-pipeline(?![\w-])/g` is word-boundary-strict (no false positive on `--test-pipeline-*` variants).
- Docs: `docs/maintainer/research-subagent-isolation-2026-04-22.md` (200 lines, public-docs-cited deep dive into the `isolation` field and its interaction with `/parallel-worktree`).
- Docs: `docs/maintainer/research-anthropic-official-2026-04-22.md` + `docs/maintainer/research-plugin-best-practice-2026-04-22.md` (preserved Codex Worker A/B parallel audit reports; basis for Phase О· / Оё / О№ / Оә planning).

## [0.1.0] - 2026-04-16

### Added
- Initial portable release split from a private project harness.
- Guardrail rules R01вҖ“R13 with TypeScript implementation in `plugins/harness/core/`.
- `harness.config.json` schema and loader for per-project customization.
- Pure-JS JSON file state store (zero native dependencies).
- Plugin manifest (`plugin.json`, `marketplace.json`) for Claude Code distribution.
- 5 verb skills: `harness-plan/work/review/release/setup`.
- 3 general agents: `worker`, `reviewer`, `scaffolder`.
- Helper agents: `security-auditor`, `codex-sync`.
- E2E smoke tests that run against compiled core without requiring a Claude Code session.
- GitHub Actions workflows for CI, build, release, and smoke testing.

### Changed
- R10 (protected directory deletion) is now parameter-driven (`protectedDirectories`), no-op when empty.
- R11 (API-key-in-command) is parameter-driven (`protectedEnvVarNames`), no-op when empty.
- All guardrail error messages are now in English and dynamically constructed from config.
- `harness-setup check` now references v3 agent names (`worker`/`reviewer`/`scaffolder`).
- State storage path unified to `.claude/state/harness.json` (JSON, not SQLite).

### Removed
- `better-sqlite3` dependency (native binary) replaced with pure JS JSON store.
- Project-specific agents (`script-tester`, `test-writer`) вҖ” see migration guide.
- Project-specific rule (`project-architecture.md`).

### Security
- Added explicit guidance on log sensitivity in `docs/en/security.md`.
- `.gitignore` template excludes `.claude/logs/`, `.claude/state/`, `.claude/worktrees/`.
