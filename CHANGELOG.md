## [Unreleased]

### Added

- **Unified exemption grammar (Phase ε) — pipe-separated 4 required fields** — `plugins/harness/core/src/__tests__/generality.test.ts` parser now enforces the form `generality-exemption: <pattern-ids> | <issue-key> | <expiry> | <reason>` for every exemption comment in shipped spec. Field contracts: `<pattern-ids>` is a CSV of `B-\d+[a-z]?` identifiers (bare `all` remains rejected per the existing Codex [C-2] guard); `<issue-key>` matches `[A-Z][A-Z0-9_]*-[A-Za-z0-9_-]+` so both numeric refs (`HARNESS-42`) and the semantic slugs required for self-reference (`HARNESS-generality-self`) and cross-project tracking (`PARTS-12`) are supported; `<expiry>` is restricted to the 3 formats enumerated in the new `CONTRIBUTING.md §3.1` field table (semver / ISO date / quarter), so the release-gate expiry requirement from [PR Review Flow §5](CONTRIBUTING.md#pr-review-flow) is now mechanically enforced rather than policy-only; `<reason>` is free text, required. Line-level comments unify to the same `generality-exemption` keyword (replacing the legacy `generality-ok`); a short form `// generality-exemption: B-1,B-2a` is accepted when the metadata is inherited from the file-head declaration. To verify the 4-field parser works end-to-end, a new `describe("exemption grammar (unified, pipe-separated)")` block in `generality.test.ts` adds 22 tests covering valid / missing-field / invalid-issue-key / invalid-expiry / legacy-form-rejection / parser-return-shape paths. `hasFileExemption` / `hasLineExemption` / the new `parseExemption` return `{ patternIds, issueKey, expiry, reason }`; `findHits` was updated to pass `pattern.id` to `hasLineExemption` so line-level exemption now requires an explicit pattern-id scope (was silently any-pattern before). Spec source: `CONTRIBUTING.md` Section 3.1 ("Unified Exemption Grammar") is new and cross-linked from Section 3 checklist and PR Review Flow. Design informed by Codex A public-docs survey (no Anthropic-official exemption guidance exists; patterned after ESLint `// eslint-disable-next-line <rule> -- <reason>` and RuboCop `# rubocop:disable Cop/Name` with stricter issue-key + expiry mandates) and Codex B independent design proposal (pipe separator chosen over em-dash + comma for parser-ambiguity elimination; em-dash + multi-comma explicitly rejected at parse-time).

### Breaking

- **Legacy exemption forms are rejected at parse-time** (Phase ε migration). The following no longer validate and cause `parseExemption` to throw, which fails `generality.test.ts` and blocks CI: (a) em dash (`—`) + comma separators such as `B-1 — HARNESS-42, v0.5.0, reason` (pipe separator is mandatory for file-level); (b) bare `generality-ok` keyword at line level (unified to `generality-exemption`); (c) `all` as a pattern-id (previously rejected, guard preserved); (d) file-level declarations with fewer than 4 pipe-separated fields (`B-1, reason only` — missing issue-key / expiry). Migration: the self-reference exemption at the head of `plugins/harness/core/src/__tests__/generality.test.ts` has been rewritten in-place to `HARNESS-generality-self | 2099-12-31 | detector harness itself must reference patterns it blocks (self-reference unavoidable, until v1.0 redesign)`. The `all` exemption previously at the top of `.github/pull_request_template.md` is removed (the template is outside `generality.test.ts` scan scope, so no replacement exemption is required; the comment now explicitly notes this). Example syntax in `docs/maintainer/english-migration.md` and `docs/maintainer/leak-audit-2026-04-22.md` is updated to the pipe form. Downstream consumer repos that shipped their own `generality-exemption` comments against this plugin must migrate to the 4-field pipe form before upgrading.

### Changed

- **`CONTRIBUTING.md` Section 3 exemption checklist item** is rewritten to cross-link the new Section 3.1 (Unified Exemption Grammar) so PR authors find the exact grammar without consulting the test source. PR Review Flow §5 wording is unchanged but now has a canonical grammar reference. `docs/maintainer/english-migration.md` Phase 4 exemption example is updated to the pipe form. `docs/maintainer/leak-audit-2026-04-22.md` mechanism section is updated to cite `CONTRIBUTING §3.1`. `.github/pull_request_template.md` drops its legacy `all` exemption (template is outside scan scope; comment now notes this explicitly).

## [0.3.3] - 2026-04-23

### Fixed

- **plugin.json manifest `hooks` field duplication — Duplicate hooks file detected (v0.3.2 regression)** — v0.3.2 `plugins/harness/.claude-plugin/plugin.json` explicitly declared `"hooks": "./hooks/hooks.json"`, but Claude Code auto-loads `hooks/hooks.json` by convention (spec: https://code.claude.com/docs/en/plugins-reference). The explicit reference triggered a runtime `Duplicate hooks file detected: ./hooks/hooks.json resolves to already-loaded file` error that blocked **all 14 hook events** (UserPromptSubmit / PostToolUseFailure / WorktreeCreate blocking protocol / WorktreeRemove / PreToolUse / PostToolUse / PermissionRequest / SessionStart / SessionEnd / PreCompact / SubagentStop / TaskCreated / Stop / TaskCompleted) from loading — the plugin was effectively silent for every project that upgraded to v0.3.2. `manifest.hooks` is reserved for *additional* hook files beyond the auto-loaded standard path; the standard `hooks/hooks.json` must be omitted from the manifest to avoid the duplicate detection. The field is now removed. Content-integrity test that previously enforced the bug (`it("hooks フィールドで hooks.json を指している", () => expect(pluginJson["hooks"]).toBe("./hooks/hooks.json"))`) is flipped to a regression guard (`hooks field が auto-load standard path (./hooks/hooks.json) を参照しない`) that rejects future reintroduction of the auto-load-path duplicate while still permitting legitimate future additional hook file references via the same field. No other shipped behavior changes; existing hook handlers remain intact and resume working after upgrade.

## [0.3.2] - 2026-04-23

### Added

- **`UserPromptSubmit` hook — Global plugin → Local project rules bridge** (observability + context injection). New `plugins/harness/core/src/hooks/user-prompt-submit.ts` implements `handleUserPromptSubmit`, registered in `plugins/harness/hooks/hooks.json` (dispatch arg `user-prompt-submit`, timeout 10s). Reads `userPromptSubmit.contextFiles` from `harness.config.json` and injects aggregated content into `hookSpecificOutput.additionalContext` so each user prompt carries project-local invariants (e.g. `.claude/rules/*.md`) without manual paste. Fail-open (config read failure / file missing / read error → silent skip + `approve`). Path-traversal guard rejects `..` segments, absolute paths, and any resolved path escaping `projectRoot` via symlink — uses `realpathSync` on both root and target plus `lstatSync.isSymbolicLink()` reject for defence-in-depth. `statSync().isFile()` gate prevents blocking reads of FIFO / socket / char-device entries. Byte-based size cap (`Buffer.byteLength`) with header pre-allocation guarantees `header + content ≤ maxTotalBytes` strictly (truncation marker is informational, counted outside the cap). Binary-safe UTF-8 truncate uses `Buffer.subarray` at byte boundary and drops invalid trailing bytes via `toString("utf-8").replace(/�+$/, "")`. Newline sanitization (`\r\n` / `\n` / `\r` → `\\n` literal) defeats fake section-boundary injection. **Per-request nonce** (12-hex-char, 48-bit entropy via `crypto.randomBytes(6).toString("hex")`) is stamped onto the open / close fence markers (`===== HARNESS PROJECT-LOCAL CONTEXT <nonce> =====` / `===== END HARNESS CONTEXT <nonce> =====`) and the truncation marker (`[harness <nonce>] context truncated at N bytes`) — an attacker cannot predict the next-request nonce (2^-48 ≈ 3.5e-15 collision probability), so literal fence markers embedded in rule-file content cannot spoof context boundaries. Spec basis: `docs/maintainer/research-anthropic-official-2026-04-22.md` § 3 (フック).
- **`PostToolUseFailure` hook — observability + corrective feedback injection** (non-blocking). New `plugins/harness/core/src/hooks/post-tool-use-failure.ts` implements `handlePostToolUseFailure`, registered in `plugins/harness/hooks/hooks.json` (dispatch arg `post-tool-use-failure`, timeout 10s). Fires when tool execution fails (exception / non-zero exit / interrupt) — mutually exclusive with `PostToolUse`. Injects tool name + sanitized error + optional corrective hint into `hookSpecificOutput.additionalContext` so Claude has next-turn recovery material without extra prompting. 6 built-in hint matchers (permission denied / no such file / command not found / signal-based abort 130/137/143 / timeout / connection refused); the signal abort regex now accepts `exit(ed)? ... (status code|status|code) N` forms to match common emitters such as `Command exited with non-zero status code 130`. Always `decision: "approve"` — tool failure itself is not blocked. Fail-open (`enabled: false` or malformed config → silent skip). Error sanitization uses the same newline escape as UserPromptSubmit plus C0 control-char escape (TAB preserved for readability), and tool-name identifier sanitization restricts to `[A-Za-z0-9_-]` with ellipsis truncation at 64 chars. **Per-request nonce** is stamped onto the header (`[harness <nonce> PostToolUseFailure] tool=...`) and the truncation marker (`[harness <nonce>] error truncated at N chars`) symmetrically with UserPromptSubmit (defence in depth against fake-marker injection from tool output).
- **14 lifecycle hook events total** (12 → 14, +`user-prompt-submit` +`post-tool-use-failure`). `plugins/harness/.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json` descriptions list the new events explicitly so Claude Code caches invalidate on upgrade.
- **Content-integrity guards** (`content-integrity.test.ts`): new `describe` blocks for both hooks (registration invariants: `hooks.json` entry + 10s timeout; handler `export` + `additionalContext` injection policy + `loadConfigSafe` fail-open; `index.ts main()` JSON-envelope dispatch with `hookSpecificOutput.hookEventName` per official spec). Signal-abort hint-coverage assertion now validates the broader regex (`exit(?:ed)?` + exit-code alternation) rather than the narrower literal.
- **Shipped-spec generality invariants** (`user-prompt-submit.test.ts` and `post-tool-use-failure.test.ts`): per-file regex guards that fail if internal tracker IDs leak into hook sources. The `post-tool-use-failure` guard mirrors the broader coding-guidelines pattern set (`PR|Issue|Task #N` / `(C-N)` / `(M-N)` / `(m-N)` / `A-N rN` / `Round N` / severity labels) so future comment drift is caught locally before plugin-wide extension.

## [0.3.0] - 2026-04-23

### Added

- **`WorktreeCreate` hook blocking protocol production (Phase κ-2)** — previously a scaffold-only handler deferred until `isolation: worktree` coordination design. Now registered in `hooks.json` (`command` type, timeout 120s) and implemented as a production blocking hook per the official contract (https://code.claude.com/docs/en/hooks): a `command` hook prints the absolute worktree path to stdout on success, and any non-zero exit causes worktree creation to fail (blocking semantics — unlike other hooks there is no special exit-2 behavior). `handleWorktreeCreate` now executes real `git worktree add <path> -b harness-wt/<name>` with: (a) strict name validation (allowlist `[A-Za-z0-9_][A-Za-z0-9_.-]{0,63}` + git refname rule enforcement: no consecutive `..`, no trailing `.`, no `.lock` suffix, no path separators / `.` prefix / shell metachars), (b) `<parent-of-cwd>/<basename-of-cwd>-wt-<name>` sibling path convention (aligns with `/parallel-worktree` manual layout when both use this prefix), (c) `harness-wt/` branch prefix to avoid namespace collision with user branches, (d) idempotent behavior (2nd call with same name + branch returns existing path via porcelain stanza parsing of both `worktree` and `branch refs/heads/...`), (e) `realpathSync.native` normalization of `projectRoot` on entry (macOS `/var` ↔ `/private/var` symlink diff and Windows 8.3 short-name `RUNNER~1` ↔ `runneradmin` both eliminated), (f) explicit split of repository presence check (`isGitRepository`) vs unborn-HEAD check (`gitHasCommit`) for clearer reason strings, (g) early newline/CR reject on `projectRoot` to prevent downstream stdout/stderr corruption, (h) post-race recheck after `git worktree add` failure to handle concurrent create. No orphan-branch fallback retry: if `-b` fails because `harness-wt/<name>` already exists, the hook surfaces the git error and the user is expected to clean up stale branches (`git branch -D harness-wt/<name>`) before retrying. `HookResult` gains optional `worktreePath` / `continue` / `stopReason` / `suppressOutput` fields (backward-compatible, used only by the new WorktreeCreate path). `index.ts main()` gains a `worktree-create` branch that writes raw stdout path on success or exits 1 on failure per the blocking contract (via `process.exitCode = 1` so stderr/stdout drain before termination). The existing `content-integrity.test.ts` Phase η P0-κ guard that asserted `WorktreeCreate` *not* registered is flipped to assert it *is* registered with `timeout: 120`, plus the scaffold self-identification guard is replaced with a production-blocking-protocol guard (requires `git worktree add` + blocking / exit / non-zero / worktreePath wording and forbids any `scaffold ... deferred` combination). `marketplace.json` plugin description is updated to **12 lifecycle hook events** (was 11). The `isolation: worktree` frontmatter is intentionally NOT added to any agent in this PR; that migration is deferred pending `/parallel-worktree` co-existence design (preventing nested-worktree interference). Test count 1836 → 1856.

Additional hardening driven by Codex second-opinion (pre-merge) review:
- **Subdirectory-safe repo root resolution**: `cwd` hook payload is Claude Code's CWD, not necessarily repo top-level. Previously a subdirectory invocation of the hook placed the sibling worktree *inside* the repo tree and broke idempotency (path+branch mismatch). Now uses `git rev-parse --show-toplevel` to resolve the actual repo root and bases `<parent>/<basename>-wt-<name>` on that.
- **`.worktreeinclude` replication**: registering `WorktreeCreate` fully replaces Claude Code's default `git worktree add`, which silently dropped copying of files listed in `.worktreeinclude` (e.g. `.env`). The handler now reads the file on success, validates each entry against path traversal (no `..`, no absolute paths, prefix-match on both repo root and worktree root), and `cpSync`s safely into the new worktree. Failures are non-fatal (worktree creation is already done) and surfaced via `[worktreeinclude-copied]` / `[worktreeinclude-skipped]` / `[worktreeinclude-failed]` notes in `additionalContext`.
- **`-z` porcelain output + prunable/existsSync filter**: `findExistingWorktree` now uses `git worktree list --porcelain -z` (NUL-separated) so paths with embedded newlines don't corrupt parsing, skips `prunable` stanzas, and verifies `existsSync(path)` before treating a stanza as a reuse candidate.

Migrating cleanly requires hook-level coordination via `WorktreeCreate` / `WorktreeRemove` (tracked as Phase κ-2, Phase 2-3 scope).
- **Phase λ — B-2f generality guard** (`generality.test.ts`). Blocks re-introduction of the predecessor-project-specific `--test-pipeline` subflow in shipped specs. Regex `/--test-pipeline(?![\w-])/g` is word-boundary-strict (no false positive on `--test-pipeline-*` variants).
- Docs: `docs/maintainer/research-subagent-isolation-2026-04-22.md` (200 lines, public-docs-cited deep dive into the `isolation` field and its interaction with `/parallel-worktree`).
- Docs: `docs/maintainer/research-anthropic-official-2026-04-22.md` + `docs/maintainer/research-plugin-best-practice-2026-04-22.md` (preserved Codex Worker A/B parallel audit reports; basis for Phase η / θ / ι / κ planning).

### Removed
