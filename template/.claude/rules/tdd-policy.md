# TDD（テスト駆動開発）ポリシー

このプロジェクトでは TDD（Test-Driven Development）を採用する。

## 基本フロー

### 🔴 Red → 🟢 Green → 🔵 Refactor

1. **Red（失敗するテストを書く）**: まず失敗するテストを書く
2. **Green（テストを通す）**: テストが通る最小限のコードを書く
3. **Refactor（リファクタリング）**: コードを改善する（テストは通ったまま）

## 絶対禁止事項

### ⛔ テストコードの書き換え禁止

**テストを通すために、テスト自体を書き換えることを禁止する。**

以下の行為は厳禁:
- テストの期待値を実装に合わせて変更する
- 失敗するテストケースを削除する
- テストの条件を緩めてパスさせる
- `skip` や `todo` でテストを無効化する（一時的なものを除く）
  - **注意**: `/work --codex` や `/breezing --codex` の tamper gate は `it.skip` を即停止扱いにする。一時 skip が必要な場合は Codex モード外で人手により対応すること

```
❌ 間違い: テストの期待値を "Hello" から "Hi" に変えてパスさせる
✅ 正しい: 実装コードを修正して "Hello" を返すようにする
```

## 例外条件

以下の場合のみ、テストコードの変更を許可する:

1. **仕様変更があった場合**: チームで合意した仕様変更に伴うテスト修正
2. **テスト自体にバグがあった場合**: テストコードのロジックに明らかな誤りがある場合
3. **可読性リファクタリング**: テストの意図は変えずに可読性のみを向上させる場合

これらの場合も、必ずコミットメッセージで理由を明記すること。

## コミットのタイミング

1. **テストを書いた時点でコミット**:
   ```
   test: {メソッド名}のテストを追加

   - 正常系・異常系を網羅
   ```
2. **テストが通った時点でコミット**:
   ```
   feat: {機能名}を実装

   - {主な変更点}
   ```
3. **リファクタリング後にコミット**:
   ```
   refactor: {対象}の可読性を向上

   - {変更内容}
   ```

## チェックリスト（実装完了時）

- [ ] 新機能に対応するテストを先に書いたか
- [ ] すべてのテストが通過しているか
- [ ] テストを書き換えてパスさせていないか
- [ ] 正常系・異常系の両方がテストされているか
